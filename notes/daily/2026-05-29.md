## Backlog deferred from tokenizer (not blocking parser)

1. **`literal` field still commented out** in `Token.__init__` (`TokenType.py:14`). NUMBER tokens have no parsed-value slot — evaluator will need to re-parse the lexeme string at execution time. Add `literal` parameter back when execution/evaluator work begins (~weeks 5–6).
2. **Lexeme is stored lowercased.** `scan()` accumulates identifiers with `curr += char.lower()`, which destroys the user's original casing. Error messages later won't be able to echo what the user typed in their original case. Fix: lowercase **only** at KEYWORDS lookup time, not during accumulation.
3. **`__repr__` dual-label confusion.** Current format reads `"Lexeme: {type.name}, lexeme: {self.lexeme}"` — both labels say "lexeme" but mean different things (type name vs. raw text). Rename first label to `"Type:"`.
4. **Cursor-model unification in `number()` and `string()`.** Both still have a vestigial `for i in range(...)` counter alongside `self.advance()`/`self.peek()`. The for-i is decorative now; switching to a `while self.peek() != "\0"`-driven loop would remove the dual-counter drift risk entirely. Optional refactor.
